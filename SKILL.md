# 清洁小车任务下发助手

**核心限制**：只能使用 `poi_search.js` 和 `map_screenshot.py` 两个脚本，**不得创建任何额外脚本文件**。

## 工作流程概述

```
用户输入（如"打扫花溪到凉亭路段"）
↓
[步骤1] 识别任务类型：
- 两点路径清扫（花溪→凉亭）
- 多点路径清扫（A→B→C→D，含途经点）
- 厕所相关清扫（需视觉识别灰色厕所图标）
- 单点周围清扫
↓
[步骤2] 地点识别与分类：
- 提取用户提到的所有地点
- 对照 known_landmarks.json 检查该地点是否在参考图上已显示：
a) 参考图已显示地标（如日圆大桥、公园西门、工会驿站、各号楼等）：查阅"已知地标区域对应速查"表直接获得区域编号，无需搜索也无需视觉定位
b) 其他地点（如凉亭、花溪、步道广场等）：需要调用 poi_search.js 搜索坐标
↓
[步骤3] 获取坐标（仅对非已知地标）：
- 如果是参考图已显示地标（已知地标区域对应速查表中存在）：无需坐标，直接查阅该表获得区域编号
- 如果是其他地点：调用 poi_search.js 搜索坐标，创建POI数据，生成地图截图
↓
[步骤4] 准备POI数据（JSON数组格式，仅对需要搜索的地点）
↓
[步骤5] 坐标匹配区域（仅对POI搜索获得坐标的地点）：
- 读取 scripts/rectangles.json 中所有39个区域的中心坐标
- 对每个POI坐标，计算其与所有39个区域中心的球面距离
- 输出距离最近的3个区域作为候选（例如：休息亭→候选R32, R33, R31）
- 将候选区域记录在案，作为后续视觉分析的锚点
↓
[步骤6] 地图分析（分情况处理）：
- 如果所有地点都是"已知地标区域对应速查"表中的地标：无需生成地图截图，直接使用表中区域编号，视觉分析仅用于规划路径
- 如果包含需要搜索的非已知地标：调用 map_screenshot.py 生成带标记的地图截图
↓
[步骤7] 视觉分析（仅用于以下目的）：
- 对于已知地标：直接使用速查表中的区域编号，无需在地图上定位地标位置
- 对于非已知地标：在截图上的候选区域中确认POI/厕所所在区域编号
- 核心目的：规划路径——确定各区域之间的连接顺序
↓
[步骤8] 规划连续路径：逐段连接所有必经点
↓
[步骤9] 逐对验证相邻性，确保无跳跃
↓
[步骤10] 生成区域可视化（重要新增步骤）：
- 确定最终的 `selected_regions` 列表（区域编号从1开始）
- 调用 visualize_regions.py 生成可视化：
  ```bash
  python scripts/visualize_regions.py <region_ids...> --screenshot tmp/regions_viz_<task_id>.png --no-open --title "清扫区域可视化"
  ```
  其中 `<region_ids...>` 是将 selected_regions 中的编号减1（因为脚本使用0-38序号）
- 示例：如果 selected_regions = [2, 3, 4, 5, 7]，则调用：
  ```bash
  python scripts/visualize_regions.py 1 2 3 4 6 --screenshot tmp/regions_viz_task_001.png --no-open
  ```
↓
输出清扫任务结果（包含可视化文件路径）
```

## 坐标匹配区域（Coordinate-to-Region Matching，关键新增步骤）

**原理**：POI搜索返回的坐标（经纬度）可以和 rectangles.json 中39个区域的中心坐标进行距离计算，精准匹配最近的区域编号。这比纯视觉估算更可靠，因为视觉分析容易因地图缩放、标签遮挡、视角偏差等因素错误估计POI所在区域。

### 执行方法

在生成地图截图前，**必须**执行以下坐标匹配计算：

```python
# 伪代码逻辑——在Shell中用python执行
python -c "
import json, math

# 读取区域数据
with open('scripts/rectangles.json') as f:
regions = json.load(f)

# POI坐标（从poi_search.js搜索结果中获取）
poi_lat, poi_lng = 24.516191, 118.17256 # 例：休息亭

def distance(lat1, lng1, lat2, lng2):
# 简化的球面距离计算（经纬度差平方和，足够精确）
return math.sqrt((lat1-lat2)**2 + (lng1-lng2)**2)

results = []
for r in regions:
rid = r['id'] + 1 # id从0开始，区域从1开始
r_lat, r_lng = r['center'][1], r['center'][0]
d = distance(poi_lat, poi_lng, r_lat, r_lng)
results.append((d, rid))

# 按距离排序，取最近5个
results.sort()
for d, rid in results[:5]:
print(f'R{rid}: 距离={d*111000:.0f}m') # 1度≈111km
"
```

### 最佳实践

1. **先算后看（Calculate First, Look Second）**：
- 先用坐标计算确定POI最近的3-5个候选区域
- 再生成地图截图，用视觉分析在候选区域中确认最终区域
- 禁止仅凭视觉估算就确定区域编号

2. **已知地标也要验证**：
- 对于参考图上已显示的地标（如各号楼、爱情湾等），查 SKILL.md 中的"楼号与清扫区域对照表"确定对应区域
- 对于搜索获得的POI（如休息亭、凉亭），必须用坐标计算匹配区域

3. **距离阈值参考**：
- 距离 < 50m → 高度确信该区域
- 距离 50-150m → 可能在该区域或相邻区域，需视觉确认
- 距离 > 150m → 不太可能在该区域，应选择更近的候选区域

4. **路径规划必须基于区域相邻性**：
- 确定每个POI所在区域后，检查两个区域是否在地理上相邻
- 如果区域不相邻，找出中间的过渡区域
- 示例：63号楼(R2)→公园入口(公园西门,R1)，路径为R2→R1（直接相邻）
- 错误示例：63号楼(R2)→错误判断为R2→R3→R4→R5→R6→R7→R39

## 详细工作流程

### 场景：路径清扫需求

**示例**：用户说"打扫花溪到凉亭路段的垃圾"

**执行步骤**：

#### 步骤1：识别路径端点
- 分析用户输入，识别出两个端点："花溪"和"凉亭"
- 确认这是一个路径清扫任务（从一个点到另一个点）

#### 步骤2：获取端点坐标

**首先检查地点是否在 known_landmarks.json 的参考图地标列表中，并查阅"已知地标区域对应速查"表**：

- **如果是参考图已显示地标且在速查表中**（如公园西门→R1、日圆大桥→R38、各号楼→对应R2-R11等）：直接查阅速查表获得区域编号，**无需搜索也无需视觉定位**
- **如果是其他地点**（如凉亭、花溪、步道广场等）：必须通过 poi_search.js 搜索坐标

**示例1**：用户说"从日圆大桥到地圆桥"

两个地点都在 known_landmarks.json 的参考图地标列表中，且均在"已知地标区域对应速查"表中：
- 日圆大桥 → 查阅速查表 → R38附近
- 地圆桥 → 查阅速查表 → R39附近
- **无需调用搜索工具，无需视觉定位**，直接使用速查表中的区域编号
- 视觉分析仅用于规划 R38→R39 之间的路径

**示例2**：用户说"打扫花溪到凉亭路段"

## 视觉分析指南

### 地图元素颜色识别（关键）

在分析地图截图时，**必须首先正确识别不同颜色的地图元素**：

| 颜色/样式 | 代表含义 | 示例 |
|---------|---------|------|
| **红色区域**（红色边界线或红底白字编号） | **39个可清扫活动区域** | 标有1-39数字的红色路段 |
| **黑色文字** | **建筑物/地标标签** | "64号楼"、"福建工会驿站"、"日圆大桥" |
| **蓝色圆点** | **POI标记点** | 搜索地点后的标记 |
| **灰色小图标** | **厕所位置** | ⛲或灰色小方块 |

**⚠️ 重要区分**：
- **红色编号区域**才是实际可清扫的路段（活动区域）
- **黑色文字**（如"xx号楼"）只是地标标签，不是清扫区域本身
- 楼号旁边的红色编号区域才是需要清扫的路段

### 如何分析地图截图

#### 正确的视觉分析步骤

**第一步：找到地标位置**
- 在地图上找到用户提到的地点标签（黑色文字，如"64号楼"、"福建工会驿站"）
- 或找到蓝色POI标记点（如果进行了POI搜索）

**第二步：识别相邻的红色清扫区域**
- 观察该地标**旁边最近的红色编号区域**是哪些
- **楼号只是参考点**：如"64号楼"旁边的红色编号区域才是清扫目标
- 根据地标的实际位置，确定它相邻或最接近的1-2个红色编号区域

**第三步：规划清扫路径**
- 根据清扫路径的起点和终点，选择**连续的红色编号区域**
- 确保选择的区域在地图上地理上连续相邻
- 两点之间通常选择3-7个连续区域

#### 具体示例

**示例1：清扫64号楼到57号楼**
- 找到"64号楼"黑色文字标签 → 观察旁边红色区域是2号区域
- 找到"57号楼"黑色文字标签 → 观察旁边红色区域是5号区域
- 清扫路径：2→3→4→5（沿路连续区域）

**示例2：清扫福建工会驿站**
- 找到"福建工会驿站"黑色文字标签（位于地图左下方）
- 观察旁边红色区域是23号区域
- 清扫区域：23号区域

**示例3：涉及厕所的清扫**
- **首先查阅"厕所预设映射表"**，检查用户提到的地标是否在预设表中
- 如果在预设表中：直接使用预设区域编号
- 如果不在预设表中：找到地标POI（蓝色圆点）或地标文字（黑色），在该地标周围搜索灰色厕所图标（⛲或灰色小方块）
- 确认灰色厕所图标**旁边的红色编号区域**
- 厕所所在区域可能与地标所在区域不同

### 区域选择策略

- **连续性优先**：尽量选择相邻的连续红色编号区域
- **覆盖完整性**：确保清扫范围覆盖用户要求的整个区域
- **合理性**：不要选择距离太远的无关区域
- **两点路径最少区域原则**：两点之间选择最少数量的区域（通常3-7个）
- **多点路径分段覆盖原则**：多点路径（3个及以上必经点）将路径拆分为多个子段，每段独立保证连续性和最少区域，合并后区域数量可放宽到10-15个
- **厕所预设优先原则**：先查阅"厕所预设映射表"，如果用户提到的地标在表中，直接使用预设区域编号。不在预设表中的厕所才通过地图上的灰色厕所图标确认

## 39个可清扫区域分布

- **北部区域**：1-10号（纬度较高）
- **中部区域**：11-25号（中间地带）
- **南部区域**：26-39号（纬度较低）
- **东部区域**：经度 > 118.17
- **西部区域**：经度 < 118.17

## 最佳实践

### 颜色识别与地标定位（最重要！）

1. **始终先进行颜色识别**：查看地图截图时，首先区分：
- 红色编号区域 = 可清扫路段（目标）
- 黑色文字 = 地标标签（参考点，不是清扫目标）
- 蓝色圆点 = POI标记
- 灰色图标 = 厕所

2. **楼号定位法（已知地标直接查速查表）**：当用户提到"xx号楼"时：
- 首先查阅"已知地标区域对应速查"表（各号楼均在表中）
- **直接使用表中的区域编号**，无需在地图上找楼号位置
- **示例**："64号楼" → 查速查表 → R2（2号路），无需在地图上找64号楼在哪
- 只有当楼号不在速查表中时，才需在地图上视觉定位

3. **地标定位法（已知地标直接查速查表）**：当用户提到地标时：
- 首先检查该地标是否在"已知地标区域对应速查"表中
- **如果在表中**：直接使用表中的区域编号，无需视觉定位
- **如果不在表中**：在地图上找到该地标的黑色文字标签，观察旁边最近的红色编号区域
- **示例**："福建工会驿站" → 查速查表 → R23（23号路），无需在地图上找工会驿站位置

### 端点识别与搜索策略

4. **明确端点识别**：准确识别路径的起点、途经点和终点。注意"经过"、"再到"、"再去"等关键词标识了必经点

5. **区分参考图地标与需搜索地标（重要）**：
- **首先检查 known_landmarks.json**：用户提到的地点是否在参考图地标列表中？
- **如果是参考图已显示地标且在"已知地标区域对应速查"表中**（如公园西门→R1、日圆大桥→R38、爱情湾→R7、各号楼→对应R2-R11等）：直接查阅速查表获得区域编号，**无需搜索、无需在地图上视觉定位**
- **如果是其他地点**（如步道广场、凉亭、花溪等）：必须通过 poi_search.js 搜索坐标并打点

6. **POI搜索策略优化**（仅对未知地点）：
- **关键词自动映射**：根据用户输入自动映射到预设搜索关键词
- 注意地名准确性：如"日圆大桥"而非"日元大桥"，确保搜索关键词正确
- **预设关键词优先**：遇到搜不到的地名，先查阅"关键词扩展策略"表中的预设关键词映射规则，严格按照表格执行，不要自行变通

7. **厕所预设映射优先策略**（重要——替代纯视觉识别）：
- **第一步**：始终先查阅"厕所预设映射表"，检查用户提到的地标是否在预设列表中
- **如果在预设表中**：直接使用预设区域编号，无需搜索坐标、无需生成地图截图、无需视觉识别灰色厕所图标
- **如果不在预设表中**：回退到原有流程——搜索附近地标获取POI，生成地图截图后通过视觉分析识别具体的厕所位置
- **回退流程**：如果用户提到"某某地点旁边的厕所"，先搜索主要地点获取POI，然后在地图截图上找到该POI旁边的灰色厕所图标
- **注意**：厕所位于预设表中的地标时，直接使用预设区域编号，不需要视觉确认

### 路径规划与验证

8. **保存POI文件**：正确格式化JSON文件供地图工具使用

9. **使用预设搜索关键词**：高德API对关键词非常敏感，遇到搜不到的情况先查"关键词扩展策略"表中的预设关键词，不要随意变通

10. **仔细视觉分析（已知地标跳过定位环节）**：对于"已知地标区域对应速查"表中的地标，直接使用表中区域编号，不要在地图上找它们的位置。视觉分析仅用于规划已知区域之间的路径顺序。对于非已知地标，认真查看地图截图上的POI标记，准确识别其在哪个区域。涉及厕所时先查"厕所预设映射表"

11. **说明选择理由**：在输出中解释为什么选择这些区域

12. **遵循检查清单**：使用"视觉分析检查清单"验证结果，特别注意颜色识别检查项

13. **最少区域原则**：两点之间默认选择最少数量的区域（3-7个）。多点路径可适当放宽（10-15个）。注意："最少"指的是最短连续路径上的最少区域，不能为了减少数量而跳过中间不相邻的区域

14. **关键词扩展策略**：当用户表达不全面时，自动扩展关键词（如"公园"→"五缘湾湿地公园"，"西门"→"湿地公园西门"）

15. **多点路径分段处理**：3个及以上必经点的路径，拆分为多个子段逐段规划，不要一次性规划整个多点路径

## 注意事项

- **POI搜索依赖高德API** - 需要网络连接，有每日调用次数限制
- **地图截图使用Playwright** - 首次运行需要下载浏览器内核
- **区域编号从1开始** - 与地图显示一致
- **视觉分析用于路径规划** - 对于已知地标（速查表中存在），直接使用区域编号，无需视觉定位；视觉分析仅用于规划区域之间的路径顺序。对于非已知地标，仍需要通过地图截图上的POI标记确认区域位置
- **不得创建额外脚本** - 只使用提供的两个工具脚本
- **文件统一保存** - 所有生成的地图截图(PNG)和POI数据文件(JSON)统一保存到 `tmp/` 目录下，永久保留，不会自动删除
- **连续任务与多点路径区分** - 用户用"再去"、"然后"、"接着"等关键词连接多个任务时，属于"连续任务"类型，输出时不包含任务之间的转场过渡区域。用户一次性描述完整清扫路径时（如"从A到B再到C"），属于"多点路径"类型，输出包含所有途经区域

## 常见错误分析与解决方案（重要）

### 错误案例分析

#### 错误0：颜色识别错误——混淆黑色文字标签与红色清扫区域
**问题**：将黑色的"xx号楼"文字标签误认为是清扫区域，或忽略了红色编号区域才是实际可清扫路段。

**示例**：用户说"打扫64号楼到57号楼"
- **错误做法**：认为64号楼是区域64，57号楼是区域57
- **正确做法**：
1. 找到"64号楼"黑色文字标签 → 观察旁边红色编号区域是2号 → 清扫起点是R2
2. 找到"57号楼"黑色文字标签 → 观察旁边红色编号区域是5号 → 清扫终点是R5
3. 路径：2→3→4→5

**根本原因**：
- 没有理解黑色文字只是地标标签，红色区域才是清扫路段
- 没有正确识别地图元素的颜色含义

**如何避免**：
- **牢记颜色规则**：红色=可清扫区域，黑色=地标标签
- **先找地标，再找区域**：先定位黑色文字标签，再观察其旁边相邻的红色编号区域
- **验证区域编号**：清扫区域的编号是1-39，楼号可能远大于39

#### 错误1：区域编号方向判断错误
**问题**：将高编号区域误认为低编号区域，或反之。
**示例**：西门(西侧)到凉亭(东北侧)，错误地选择了1-13号区域，实际应选择30-39号区域。

**根本原因**：
- 没有仔细观察POI标记在地图上的实际位置
- 混淆了区域编号与地理方位的对应关系

#### 错误2：选择过多无关区域
**问题**：选择了大量与两点路径无关的区域。
**示例**：中国第一爱情湾到公园入口，错误地选择了12个区域，实际只需要3个区域(31、32、39)。

**根本原因**：
- 没有遵循"最短路径"原则
- 误将"沿湖所有区域"当作"两点之间区域"
- 没有先定位起点和终点所在的具体区域

#### 错误3：选择距离POI标记过远的区域
**问题**：当端点坐标离可选路段较远时，错误地选择了编号对应的区域，而不是距离POI标记最近的实际区域。
**示例**：公园西门到凉亭，错误地选择了1号区域(在左上角)，实际凉亭在右侧，应选择37号区域。

**根本原因**：
- 仅凭区域编号猜测，没有观察POI标记与区域的空间关系
- 没有遵循"距离POI标记最近"的原则

#### 错误4：POI标记与区域对应关系判断错误
**问题**：将POI标记与错误的区域编号对应，没有准确判断POI标记所在的具体区域。
**示例**：爱情湾标记明显在7号区域旁边，但错误地判断为5号区域。

**根本原因**：
- 没有仔细观察POI标记与区域编号的精确对应关系
- 凭感觉猜测，没有使用"视觉检查清单"验证
- 没有遵循"先定位POI标记，再找最近区域"的原则

#### 错误5：搜索关键词不精确
**问题**：当用户表达不全面时，搜索关键词不够精确，导致无法找到正确结果或找到错误结果。
**示例**：用户说"公园西门"，但实际上是指"五缘湾湿地公园(西门)"，直接使用"公园西门"搜索可能找到不相关结果。

**根本原因**：
- 直接使用用户输入的关键词，没有进行扩展或补充
- 没有考虑五缘湾湿地公园的上下文
- 没有尝试多个相关关键词组合

#### 错误6：未使用 tmp/ 目录保存生成文件
**问题**：POI数据文件和地图截图直接保存在了技能根目录下，没有使用统一的 `tmp/` 目录。

**正确做法**：
- POI数据文件 → `tmp/poi_*.json`（如 `tmp/poi_lovebay.json`）
- 地图截图文件 → `tmp/map_*.png`（如 `tmp/map_lovebay.png`）
- 所有生成的文件必须统一保存在 `tmp/` 目录下，不要放在技能根目录

#### 错误7：最短路径判断错误
**问题**：在判断两点间最短路径时，选择了错误的区域序列。
**示例**：爱情湾到感恩广场，错误地包含了38号区域，或错误地将39号放最后，正确路径应为7→39→32→31→30→29→28→27→26（39号地圆桥在7号与32号之间，路径必须严格保持地理连续性）。

**根本原因**：
- 没有仔细观察两点之间的实际地理路径
- 没有遵循"最短路径优先"原则
- 对区域分布理解不准确，凭感觉选择区域

#### 错误8：路径区域顺序排列错误
**问题**：找对了区域编号，但排列顺序错误，将中间的区域放到路径末尾，导致清扫路线不连续。
**示例**：爱情湾(7)→感恩广场(26)，正确顺序为7→39→32→31→30→29→28→27→26，但错误地写成7→32→31→30→29→28→27→26→39（将39号放到了末尾）。

**根本原因**：
- 没有验证区域编号的地理排列顺序是否与路径方向一致
- 没有在列表中按从起点到终点的实际经过顺序排列区域
- 跳过了"路径顺序"检查清单项

**如何避免**：
- 列出区域后，对照地图确认每个区域在路径上的先后顺序
- 从起点区域开始，沿着路径方向逐个列出经过的区域，直到终点区域
- 使用检查清单中的"路径顺序"项进行最终验证

#### 错误9：跳跃性选区——遗漏路径中间区域
**问题**：在多点路径清扫任务中，选择了非相邻的区域作为连续路径，跳过了中间必须经过的区域，导致清扫路线在地理上出现"跳跃"。这通常发生在只确认了POI所在的端点区域、却没有检查这些区域之间是否直接相邻的情况下。
**示例**：用户说"打扫日圆大桥旁边厕所到地圆桥旁边厕所，经过爱情湾"。根据"厕所预设映射表"和"已知地标区域对应速查"：
- 日圆大桥旁边厕所 → **R15**（起点）
- 爱情湾 → **R7**（途经点，"经过"标识）
- 地圆桥旁边厕所 → **R33**（终点）

错误做法：R15→R33→R7（把途经点爱情湾放到了终点之后，且跳过中间区域）
正确顺序：**起点(R15)→途经点爱情湾(R7)→终点(R33)**

即：按用户表述"从A到B，经过C"，顺序应为 **A(起点)→C(途经点)→B(终点)**。先到R15→通过中间连续区域到达R7(爱情湾)→再通过中间连续区域到达R33(地圆桥厕所)。所有区域必须逐对相邻，不能跳跃。

**根本原因**：
- 没有逐对验证所选区域是否地理相邻——以为"起点区域→终点区域"可以直接到达
- 在多点路径中只确认了POI归属区域，忽略了POI之间的过渡区域是否连续
- 误解了"最少区域原则"——以为跳过中间区域可以减少数量，实际上最短路径必须包含所有中间连续区域
- 没有沿着地理走向，逐个检查从起点区域出发下一个会到达哪个相邻区域

**如何避免**：
- 对路径中的每一对相邻区域（第N个和第N+1个），逐对验证它们是否地理上直接相邻
- 如果两个区域不相邻，找出它们之间所有中间区域并依次补全
- 确认最终的区域列表中从第一个到最后一个，每个区域都与前后区域相邻
- "最少区域"是指最短连续路径上的最少区域，而不是跳过中间区域

#### 错误10：纯视觉估算误差——忽略坐标匹配导致区域判断错误（重要！）
**问题**：仅凭视觉分析估算POI所在的区域编号，导致POI与区域对应错误。这是最常见且最严重的问题之一。

**典型错误案例**：
- 休息亭坐标[118.17256, 24.516191]被错误判断为R17，实际为R32。R17中心(118.16701, 24.51653)与休息亭经度差0.00555（约620米），而R32中心(118.17235, 24.51577)经度差仅0.00021（约23米）。
- POI位于两块区域交界处时，没有候选列表，凭感觉选了一个不相关的区域

**根本原因**：
- 高德地图截图上的红色区域编号文字可能被遮挡、重叠，导致视觉误判
- 地图缩放比例造成的视觉错觉——看似近的区域实际距离较远
- 没有利用 rectangles.json 的区域中心坐标数据做距离计算
- 违反了"先算后看"原则——跳过坐标计算，直接进行视觉分析

**如何避免**：
- **强制坐标匹配**：只要有POI坐标，就必须计算其与所有39个区域中心的距离
- **候选列表法**：取距离最近的3个区域作为候选，视觉分析时在候选列表中确定最终区域
- **双重验证**：坐标匹配结果和视觉分析结果必须吻合，如有冲突以坐标匹配为准
- **交界处处理**：POI位于两块区域交界处时，两块区域都纳入路径中
- **阈值参考**：距离<50m可直接确定区域，>150m说明该区域可能不匹配

#### 错误11：路径选择策略——优先短路径而非沿湖长路径
**问题**：当起点和终点分别在地图两侧时，错误选择了沿湖绕行的长路径，忽略了通过地图中部主干道的短路径。

**示例**：用户说"打扫日圆大桥旁边厕所到地圆桥旁边厕所，经过休闲平台"
- **错误做法**：选择沿湖绕行路径 R15→R16→R17→...→R33（19个区域），绕湖一整圈
- **正确做法**：选择通过日圆大桥主干道的短路径 R15→R14→R38(日圆大桥)→R30(休闲平台)→R31→R32→R33（7个区域）

**数据分析**：
- R38区域边界东端(118.169893)与R30区域边界西端(118.169755)在经度上有重叠
- R38南部与R30北部纬度间隔仅约63m
- 沿湖路径19个区域 vs 短路径7个区域，效率差距63%
- 清扫车可沿日圆大桥主干道直接从R38向南/北连接R30休闲平台，无需绕湖一整圈

**根本原因**：
- 惯性思维选择了沿湖环绕的路径，没有考虑地图内部是否存在更短的连接
- 没有充分分析区域边界的重叠情况，错误认为所有路径必须沿湖走
- 没有意识到R38（日圆大桥）作为交通枢纽，可以连接地图西侧(R15附近)和南侧(R30休闲平台附近)

**如何避免**：
- 在规划路径时，先观察整个地图的交通枢纽（如日圆大桥、地圆大桥）是否可以作为捷径
- 分析各区域边界是否在经纬度上有重叠——即使中心点距离较远，边界重叠区域也可直接连通
- 沿湖路径不是默认选项，只有当用户明确要求"沿湖"、"环湖"时才使用
- 默认优先选择经过地图中部主干道的最短路径
- 具体参考：R15(日圆大桥旁厕所)→R14→R38(日圆大桥)→R30(休闲平台)→R31→R32→R33(地圆桥旁厕所) 比沿湖线节省63%区域

[步骤7] 验证每对相邻红色区域是否地理相邻
↓
[步骤8] 验证顺序是否从起点到终点依次排列
↓
输出结果
```

### 关键原则

#### 已知地标速查原则（新增——替代视觉定位）
0. **已知地标直接查表**：对于"已知地标区域对应速查"表中的地标（各号楼、公园西门、爱情湾、日圆大桥、地圆桥、休闲平台等），直接使用表中的区域编号，**无需在地图上找它们的位置**
1. **视觉仅用于规划路径**：当所有必经点都是已知地标时，视觉分析仅用于规划区域之间的连接顺序，不是用于定位地标本身
2. **混合情况处理**：当已知地标和需搜索地标混合时，已知地标用速查表、非已知地标用视觉定位，然后用所有区域编号一起规划路径

#### 颜色识别原则（最重要！）
3. **红色=清扫区域**：地图上红色边界线或红底白字编号标记的才是39个可清扫活动区域
4. **黑色=地标标签**：黑色文字（如"64号楼"、"福建工会驿站"）只是建筑物或地标名称标签，不是清扫区域
5. **先找地标，再找区域**：先定位黑色文字标签位置（仅对非已知地标），再观察其旁边相邻的红色编号清扫区域
6. **楼号≠区域号**：用户说"64号楼"时，清扫的是该楼旁边相邻的红色编号区域（如2号），不是区域64

#### 路径规划原则
7. **先算后看（最重要！）**：先通过坐标匹配计算确定POI的候选区域，再用视觉分析验证。禁止跳过坐标计算直接进行视觉估算。
8. **先定位，再找路径**：必须先确定起点和终点各自在哪个红色编号区域，再找它们之间的路径
9. **距离最近优先**：选择距离地标最近的红色编号区域作为起点/终点
10. **最短路径优先**：默认选择两点之间的最短直线路径，除非用户明确要求"沿湖"、"环湖"等
11. **最少区域原则**：在满足覆盖的前提下，选择最少数量的区域（通常3-7个）
12. **验证连续性**：确保每对前后相邻的区域在地理上都直接相邻。不能从区域A直接跳到区域C——必须找到中间区域B并包含进来
13. **最少≠跳跃**："最少区域原则"指的是最短连续路径上的最少区域，而不是跳过中间不相邻的区域
14. **路径顺序正确**：区域列表必须按从起点到终点的实际经过顺序排列

#### 任务处理原则
15. **多点路径分段规划**：当用户需求包含3个及以上必经点时，将路径拆分为多个子段，逐段连接，再合并去重
16. **厕所预设优先**：先查阅"厕所预设映射表"，如果用户提到的地标在表中，直接使用预设区域编号。不在预设表中的厕所才通过地图截图上的灰色厕所图标确认
17. **途经点识别关键词**：注意用户表述中的"经过"、"途经"、"顺路"、"再到"、"再去"等关键词
18. **连续任务与多点路径区分**：用户用"再去"、"然后"等连接多个独立任务时，每个任务独立输出；一次性描述完整路径时，合并所有段

## 常见问题排查

### POI标记不显示
- 检查POI文件格式是否为**直接数组**，不是嵌套对象 `{ "pois": [...] }`
- 检查是否使用了正确的工作目录（必须在car_tool目录下执行）
- 检查坐标是否在config.json定义的bounds范围内

### 地图范围不正确
- 地图范围固定使用 `scripts/config.json` 中的bounds配置
- 不会根据POI位置自动调整，确保POI在配置的范围内
